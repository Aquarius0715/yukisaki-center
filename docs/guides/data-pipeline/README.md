# データ収集・前処理 実装ガイド

## このガイドの目的

雪道安全経路案内システムのうち、最初に実装するデータ収集・前処理基盤の手順をまとめる。対象は長岡市周辺のAWSハッカソン向けMVPである。

結論として、次の順に作る。

1. 対象範囲、データ契約、S3キーを固定する。
2. OpenStreetMapの道路データを一度だけ取り込み、道路区間を生成する。
3. 標高を付与し、勾配を計算する。
4. 気象データの原本保存と正規化を、既定停止のEventBridge経由で実行する。
5. 消雪パイプと除雪車GPSの仮データを同じ契約で処理する。
6. ローカルで再現できた処理をAWSへ載せる。
7. 最後にPostGISへロードし、指数計算へ引き渡す。

最初から全サービスをAWSへ作らない。まず小さな対象範囲と固定データで、`raw -> normalized -> curated` が一周することを確認してから定期実行へ進む。

## MVPの最短マイルストーン

### M0: 契約を決める

- 対象範囲を1つのGeoJSONポリゴンで管理する。
- 時刻は保存時にUTC、表示時にJSTとする。
- 座標参照系は保存・交換時に`EPSG:4326`とする。
- 原本を上書きしないS3キーを決める。
- 各出力に`run_id`、`source`、`source_urls`、`fetched_at`、`target_start_at`、`target_end_at`、`checksum_sha256`、`is_simulated`を持たせる。

### M1: 静的パイプライン

OSM原本から道路区間を作り、標高・勾配を付与したGeoParquetまたはGeoJSONをS3へ出力する。ここではRDSを必須にしない。

### M2: 動的パイプライン

気象データを定期取得し、原本と正規化JSON/ParquetをS3へ保存する。同じ入力を再処理しても同じ結果になり、重複しないことを確認する。

### M3: 仮データ

消雪パイプCSVと除雪車GPS JSON Linesを投入し、`segment_id`への関連付け結果を出力する。

### M4: DBロード

検証済みの道路区間、最新気象、仮設備・除雪履歴をPostGISへUPSERTする。指数計算はこの後の別工程とする。

## 最初の1週間の例

| 日 | 作業 | 完了状態 |
|---|---|---|
| 1 | AOI、スキーマ、S3キー、サンプル入力を決定 | サンプルで契約テストが通る |
| 2 | OSM取得と道路抽出 | 対象道路が地図で確認できる |
| 3 | 道路区間分割とID生成 | 同じ入力から同じIDが生成される |
| 4 | 標高取得と勾配計算 | 欠損率と異常値をレポートできる |
| 5 | 気象原本の取得と正規化 | 取得時刻・データ時刻を区別して保存できる |
| 6 | AWSのS3、Lambda/Fargate、EventBridge、監視を構築 | 既定停止と開発時の有効化を確認できる |
| 7 | 仮データ、再実行、障害シナリオを確認 | 直前正常データへ戻せる |

## この段階では作らないもの

- Bedrockによる説明生成
- Web APIと画面
- 経路探索API
- リアルタイムストリーミング基盤
- 機械学習モデル
- 本番相当のマルチAZ・自動スケーリング

これらを後回しにしても、収集原本と正規化データの契約を保てば後から接続できる。
