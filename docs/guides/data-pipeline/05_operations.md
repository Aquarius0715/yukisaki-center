# 運用・障害対応手順

## 1. 実行manifest

すべての処理は次のようなmanifestを残す。

```json
{
  "run_id": "uuid",
  "pipeline": "weather-forecast",
  "schema_version": "1.0.0",
  "status": "succeeded",
  "started_at": "2026-07-12T00:00:00Z",
  "finished_at": "2026-07-12T00:00:05Z",
  "input_keys": ["s3://..."],
  "output_keys": ["s3://..."],
  "input_count": 1,
  "output_count": 20,
  "quarantine_count": 0,
  "checksum_sha256": "...",
  "code_version": "git-sha-or-image-digest",
  "quality": {
    "missing_ratio": 0.0,
    "warnings": []
  }
}
```

失敗時もmanifestを残し、`status`、エラー分類、最後に成功したrunを記録する。スタックトレースや外部レスポンス全体はCloudWatch Logs側へ送り、manifestを巨大にしない。

## 2. 構造化ログ

ログには次を含める。

- `timestamp`
- `level`
- `pipeline`
- `run_id`
- `dataset`
- `stage`
- `duration_ms`
- `record_count`
- `error_code`
- `retry_count`

車両ID、座標、外部レスポンス、秘密値を無条件にINFOログへ出さない。

## 3. メトリクスとAlarm

| メトリクス | 初期Alarm条件例 |
|---|---|
| 収集成功数 | 予定時間内に0回 |
| 連続失敗数 | 2回以上 |
| データ鮮度 | 気象の最新正常データが2時間超 |
| 取得件数 | 前回比で大幅減少 |
| quarantine件数 | 1件以上、または割合が閾値超過 |
| 処理時間 | 通常値の2倍超が継続 |
| Lambda Error/Throttle | 1件以上を調査 |
| EventBridge target DLQ | メッセージ1件以上 |
| ECS Task失敗 | 終了コード非0、または品質検査失敗 |
| RDS容量・接続数 | 設定閾値超過 |

最初の数日は観測値を集め、誤通知が多すぎない現実的な閾値へ調整する。

## 4. 再実行手順

1. 失敗した`run_id`とstageを特定する。
2. rawが正常に保存されているか確認する。
3. rawが正常なら外部へ再アクセスせず、そのS3キーから後段だけ再実行する。
4. rawが壊れている場合だけ、利用条件と取得間隔を守って再取得する。
5. 新しい`run_id`を発行し、`replay_of_run_id`に元runを記録する。
6. 品質検査後にcuratedを公開する。
7. DBロード済みなら、同一自然キーへの冪等なUPSERTであることを確認する。

過去rawをコード新版で再処理する場合は、`code_version`と`schema_version`を更新し、旧curatedを削除せず新しいsnapshotとして出力する。

## 5. 代表的な障害対応

### 外部データ取得失敗

- タイムアウト、DNS、429、5xxを区別する。
- 規定回数だけ指数バックオフで再試行する。
- 最後の正常curatedを維持し、`latest`を失敗runへ向けない。
- 最新正常データの時刻をAPIへ渡し、2時間超なら「情報が古い」と表示可能にする。

### 外部スキーマ変更

- rawは保存する。
- 解析失敗データをquarantineへ移す。
- normalized/curated/DBを更新しない。
- fixtureを追加してadapterを修正する。
- 旧形式と新形式の両方の契約テストを通してから再処理する。

### 道路更新でsegment_idが変わる

- OSM版と道路生成コード版を確認する。
- 意図しないID変化なら公開しない。
- 正当な道路変更なら旧新ID対応表を生成する。
- 消雪パイプ仮データ等を新IDへ移行し、未対応をquarantineへ送る。
- 経路グラフの接続性テスト後に切り替える。

### 標高欠損・異常勾配

- 使用DEMとタイル欠損を確認する。
- より低精度なDEMへのフォールバック結果を確認する。
- 橋梁、トンネル、短区間、DEM境界を確認する。
- 推測値で埋めず、欠損率と注意フラグを残す。

### GPSが道路へ結合できない

- AOI、検索半径、座標順序、時刻順、異常速度を確認する。
- 高架・並行道路では最近傍だけで確定しない。
- 未結合を除雪済みとして扱わない。
- `unmatched`割合をデモ品質レポートへ出す。

## 6. リリースチェックリスト

- [ ] 対象データの利用条件と出典表示を再確認した。
- [ ] rawからcuratedまで追跡できる。
- [ ] 同一入力の再実行で重複しない。
- [ ] fixture、境界値、欠測、仕様変更テストが通る。
- [ ] 仮データに`is_simulated: true`が付く。
- [ ] 不正入力がquarantineへ送られる。
- [ ] 最後の正常データを維持できる。
- [ ] Schedulerの再試行とDLQを確認した。
- [ ] CloudWatch Alarmの通知を確認した。
- [ ] S3、Secrets Manager、RDSが公開されていない。
- [ ] RDSロード前の品質ゲートが機能する。
- [ ] 主要地点間の道路グラフが接続している。
- [ ] データ時刻、取得時刻、処理時刻を区別できる。

## 7. 収集・前処理フェーズの受入条件

| ID | 条件 |
|---|---|
| DP-01 | OSM原本、取得情報、チェックサムがS3 rawに保存される |
| DP-02 | AOI内の車両向け道路が安定した`segment_id`へ分割される |
| DP-03 | 道路区間に標高、最大勾配、標高ソース、欠損率が付く |
| DP-04 | 気象原本が定期取得され、観測/予報時刻と取得時刻を区別できる |
| DP-05 | 外部取得失敗時に直前正常データを維持する |
| DP-06 | 気象を道路へ結合した距離または区域関係が記録される |
| DP-07 | 消雪パイプ仮データの参照整合性と値域が検証される |
| DP-08 | 除雪車GPSの結合結果と未結合理由が記録される |
| DP-09 | 同一rawから同じcuratedを再生成できる |
| DP-10 | 不正入力でDBと`latest`が更新されない |
| DP-11 | run単位の件数、処理時間、品質、コード版を追跡できる |
| DP-12 | 公開データの出典と仮データ使用を下流へ引き渡せる |

これらを満たした後、走りやすさ指数のルール実装へ進む。
